# 切り替え式の最強版レビュー Prompt

以下をそのままコピペして使ってください。  
先頭の `MODE` や `DEPTH` だけ変えれば、`デバッグレビュー / リファクタリング / 新機能提案 / 全部入り` を切り替えられます。

---

## Prompt

```md
以下のプロジェクトをレビューしてください。

## Review Settings
- MODE: `all`
- DEPTH: `deep`
- OUTPUT_LANGUAGE: `ja`
- FINDINGS_FIRST: `true`
- INCLUDE_FILE_REFERENCES: `true`
- INCLUDE_TEST_PLAN: `true`
- INCLUDE_IMPLEMENTATION_IDEAS: `true`

### MODE の意味
- `debug`: デバッグ・不具合・将来エラー・整合性・性能劣化の観点を最優先
- `refactor`: リファクタリング・コード整理・命名改善・型安全化・関数分割・重複排除を最優先
- `feature`: 新機能提案・UX改善・拡張案を最優先
- `all`: 上記を全部やる

### DEPTH の意味
- `quick`: 高リスク箇所中心に短く鋭く見る
- `standard`: バランス良く全体を見る
- `deep`: 広く深く監査し、根拠付きで詳しく出す

## あなたの役割
あなたは、実務レベルでコードベースを診断するシニアエンジニアです。
表面的な感想ではなく、実際に保守・運用・機能追加で効くレビューをしてください。

## 必須ルール
- まずコードベースを探索し、事実を集めてから判断する
- 推測より実際の構成・依存関係・設定・主要導線・型・API を優先する
- AGENTS.md やローカルルールがあれば必ず従う
- 実行可能なら lint / build / test を確認し、その結果も根拠に含める
- 「なんとなくよくない」ではなく、何がどう危険かを明示する
- 可能ならファイルパスと行番号を示す
- 抽象論ではなく、直し方や分割方針まで具体化する

## まず最初にやること
1. プロジェクト構成を把握する
2. 主要なエントリポイント、画面、API、状態管理、ユーティリティ、型定義を確認する
3. 使用技術と依存関係を整理する
4. lint / build / test の有無と現状を確認する
5. 重いファイル、責務過多のファイル、危険そうな境界を特定する

## MODE 別レビュー観点

### MODE=`debug` または `all`
以下を重点的に確認してください。
- 実行時エラーの可能性
- null / undefined / 非 null アサーションの危険性
- 非同期処理の競合、二重送信、キャンセル漏れ、状態不整合
- API 入力検証不足、レスポンス shape の不統一
- DB 書き込み時の途中失敗、不整合、削除事故、トランザクション不足
- 認証 / 認可 / 権限過多 / secret 露出リスク
- 本番運用で危険な箇所
- パフォーマンス問題
  - 不要な再描画
  - N+1 fetch
  - 重い client component
  - キャッシュ不足
  - 画像最適化不足
  - 無駄な初期 fetch

### MODE=`refactor` または `all`
以下を重点的に確認してください。
- 責務過多のコンポーネント / 関数 / route handler
- 分割した方がよい state / side effect / UI / data access
- 命名の曖昧さ、不統一、責務とのズレ
- 重複コード、共通化余地、helper 化余地
- 型安全性の弱い箇所
  - any
  - 暗黙キャスト
  - 雑な `as`
  - 境界の型不足
- テストしづらい構造
- 変更容易性を落としている構造的問題

### MODE=`feature` または `all`
以下を重点的に確認してください。
- 既存プロダクトと相性が良い新機能
- UX 改善
- 継続利用を促す改善
- AI 活用の自然な拡張
- PWA / offline / 通知 / 自動化の余地
- 既存データを活かせる提案
- 低コスト高効果の案
- 中長期で効く案

## 出力ルール

### 1. まず最初に「事実ベースの現状整理」
- 技術スタック
- 現在の品質ゲート状況（lint / build / test）
- 主要な責務の分布
- 高リスク領域

### 2. FINDINGS_FIRST=true の場合
最初に Findings を出してください。
重要度順に並べ、各 finding に以下を含めてください。
- 何が問題か
- なぜ危険か
- どこで起きるか
- 発火条件や影響
- 推奨修正方針

### 3. MODE に応じて以下を出す

#### debug / all
- バグ・将来リスク・整合性・性能改善のレビュー
- 今すぐ直すべき項目
- 後回しでもよい項目

#### refactor / all
- リファクタリング提案
- 関数分割 / コンポーネント分割案
- 命名改善案
- 型安全化案
- 重複排除 / 共通化案

#### feature / all
- 新機能提案を 5〜10 件
- 各案に以下を含める
  - 機能名
  - 何が便利になるか
  - ユーザー価値
  - 実装難易度
  - MVP の切り方
  - 今の構成との相性

### 4. 最後に実行計画
- 優先順位付きの実施順
- 低コスト高効果
- 高コスト高効果
- 追加すべきテスト
- 先に整えるべき技術的土台

## 出力フォーマット
- 日本語
- 実務で使える粒度
- 根拠付き
- 可能ならファイルパスと行番号付き
- 箇条書きに頼りすぎず、要点を整理して読みやすく出す
- 問題が無ければ「重大な finding はなし」と明記し、残留リスクを書く
```

---

## 使い方の例

### 1. バグ・事故予防に全振り
```md
- MODE: `debug`
- DEPTH: `deep`
```

### 2. コード整理に全振り
```md
- MODE: `refactor`
- DEPTH: `deep`
```

### 3. 新機能アイデア出しに全振り
```md
- MODE: `feature`
- DEPTH: `standard`
```

### 4. 全部入りのフル監査
```md
- MODE: `all`
- DEPTH: `deep`
```

## おすすめ設定
- まずは `MODE=all` `DEPTH=deep`
- 時間が無いときは `MODE=debug` `DEPTH=quick`
- 実装前の整理だけしたいときは `MODE=refactor`
- 次の開発テーマを決めたいときは `MODE=feature`
